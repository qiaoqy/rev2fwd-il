## 1. Experiment Overview

### 1.1 Goal

Demonstrate that the Rev2Fwd method generalizes beyond the Piper arm by:
1. Taking **existing Task B demonstrations** (Inovo arm, collected by TCL team)
2. **Time-reversing** them to create Task A training data
3. **Training a DiT Flow policy** on the reversed data
4. **Evaluating & iterating** via DAgger (in simulation or on real hardware)

### 1.2 Task: Tower Body Pick & Place

The initial dataset is `0209_tower_boby` — picking up a purple disc from a pole and placing it on a sponge.

| Property | Value |
|----------|-------|
| **Task description** | "Pick up the purple disc from the pole and place it on the sponge." |
| **Robot** | Inovo 6-DoF arm |
| **Cameras** | 2× RealSense (primary + gripper-mounted), 848×480 resolution |
| **Episodes** | 51 demonstrations (~500-660 frames each at ~30 Hz) |
| **Data size** | ~7.5 GB (uncompressed) |
| **Data format** | Per-frame NPZ files (RoboKit `DataHandler` format) |

### 1.3 Task Direction Convention

| Direction | Description | Data Source |
|-----------|-------------|-------------|
| **Task B (original)** | Pick disc from pole → Place on sponge | Existing `0209_tower_boby` demos |
| **Task A (reversed)** | Pick disc from sponge → Place on pole | Generated by time reversal |

## 3. Data Format Specification

### 3.1 Per-Frame NPZ Structure

Each `.npz` file contains one timestep, serialized via [RoboKit DataHandler](https://github.com/ygtxr1997/RoboKit/blob/dev/src/robokit/data_manager/data_handler.py).

| Field | Raw Type | Decoded Type | Shape | Description |
|-------|----------|--------------|-------|-------------|
| `primary_rgb` | JPEG bytes | `uint8` ndarray | `(480, 848, 3)` | Primary (fixed) camera RGB |
| `gripper_rgb` | JPEG bytes | `uint8` ndarray | `(480, 848, 3)` | Gripper-mounted camera RGB |
| `primary_depth` | PNG bytes | `uint16` ndarray | `(480, 848)` | Primary camera depth map |
| `gripper_depth` | PNG bytes | `uint16` ndarray | `(480, 848)` | Gripper camera depth map |
| `language_text` | pickle → `np.array(str)` | `str` | scalar | Natural language task description |
| `actions` | pickle → `float64` | `float64` ndarray | `(7,)` | Velocity command (see §3.3) |
| `rel_actions` | pickle → `float64` | `float64` ndarray | `(7,)` | Same as `actions` (identical) |
| `robot_obs` | pickle → `float64` | `float64` ndarray | `(14,)` | Robot observation (see §3.2) |
| `force_torque` | pickle → `float64` | `float64` ndarray | `(6,)` | Force/torque sensor `[fx, fy, fz, mx, my, mz]` |

### 3.2 `robot_obs` (14D) Breakdown

| Index | Field | Unit | Description |
|:-----:|-------|------|-------------|
| 0–2 | `tcp_pos` | meters | TCP position `[x, y, z]` in robot base frame |
| 3–5 | `tcp_ori` | radians | TCP orientation `[roll, pitch, yaw]` (Euler RPY) |
| 6 | `gripper_width` | [0, 1] | Normalized gripper opening (0=closed, 1=fully open) |
| 7–12 | `joint_states` | radians | 6 joint angles |
| 13 | `gripper_action` | {0, 1} | Binary gripper command (0=open, 1=closed) |

**Typical values observed**:
- `tcp_pos`: x ∈ [-0.05, 0.10], y ∈ [0.35, 0.45], z ∈ [0.35, 0.55]
- `tcp_ori`: roll ≈ ±π, pitch ≈ 0, yaw ∈ [-0.05, 0.15]
- `gripper_width`: 0.51 (closed on object) to 0.85 (open)

### 3.3 `actions` (7D) — Velocity Commands

| Index | Field | Unit | Description |
|:-----:|-------|------|-------------|
| 0–2 | `v_xyz` | m/s | TCP linear velocity `[vx, vy, vz]` |
| 3–5 | `v_rpy` | rad/s | TCP angular velocity `[v_roll, v_pitch, v_yaw]` |
| 6 | `gripper` | {0, 1} | Binary gripper command |

**CRITICAL**: Actions are **velocity commands** at ~30 Hz, NOT position deltas:

```
obs[t+1].tcp_pos ≈ obs[t].tcp_pos + (1/30) × actions[t].v_xyz
```

Verified empirically: `dt ≈ 0.033s` gives mean position prediction error < 0.4mm.

**Action statistics**:
- XYZ velocity range: [-0.10, +0.10] m/s
- RPY velocity range: [-0.88, +0.74] rad/s
- Gripper: binary {0, 1}

### 3.4 Decoding NPZ Files

```python
import numpy as np
import pickle
import cv2

def load_frame(path: str) -> dict:
    """Load and decode a single RoboKit NPZ frame."""
    f = np.load(path, allow_pickle=True)

    # Images: stored as encoded bytes (JPEG for RGB, PNG for depth)
    primary_rgb = cv2.imdecode(
        np.frombuffer(f['primary_rgb'].item(), np.uint8),
        cv2.IMREAD_COLOR
    )  # (480, 848, 3) uint8, BGR format
    primary_rgb = cv2.cvtColor(primary_rgb, cv2.COLOR_BGR2RGB)

    gripper_rgb = cv2.imdecode(
        np.frombuffer(f['gripper_rgb'].item(), np.uint8),
        cv2.IMREAD_COLOR
    )
    gripper_rgb = cv2.cvtColor(gripper_rgb, cv2.COLOR_BGR2RGB)

    # Numeric fields: stored as pickled numpy arrays
    robot_obs   = pickle.loads(f['robot_obs'].item())     # (14,) float64
    actions     = pickle.loads(f['actions'].item())        # (7,)  float64
    rel_actions = pickle.loads(f['rel_actions'].item())    # (7,)  float64
    force_torque = pickle.loads(f['force_torque'].item())  # (6,)  float64
    language    = pickle.loads(f['language_text'].item())   # str

    return {
        'primary_rgb': primary_rgb,
        'gripper_rgb': gripper_rgb,
        'robot_obs': robot_obs,
        'actions': actions,
        'rel_actions': rel_actions,
        'force_torque': force_torque,
        'language_text': language,
    }
```

## 7. Script 2: Time Reversal

### 7.1 Purpose

Generate Task A training data by time-reversing the original Task B demonstrations. This is the **core of the Rev2Fwd method**.

### 7.2 Planned Script: `2_time_reverse.py`

### 7.3 Reversal Algorithm

For each episode with T frames:

```
Original (Task B):  frame[0], frame[1], ..., frame[T-1]
Reversed (Task A):  frame[T-1], frame[T-2], ..., frame[0]
```

**Per-frame reversal**:

| Field | Reversal Rule |
|-------|---------------|
| `primary_rgb` | Reverse frame order |
| `gripper_rgb` | Reverse frame order |
| `primary_depth` | Reverse frame order |
| `gripper_depth` | Reverse frame order |
| `robot_obs[0:6]` (tcp pose) | Reverse frame order |
| `robot_obs[6]` (gripper width) | Reverse frame order |
| `robot_obs[7:13]` (joints) | Reverse frame order |
| `robot_obs[13]` (gripper action) | Reverse AND invert: `1 - original` |
| `language_text` | Update to describe reversed task |
| `force_torque` | Reverse frame order |
| **`actions[0:6]` (velocities)** | **Negate AND reverse**: `-actions[T-1-t][0:6]` |
| **`actions[6]` (gripper)** | **Reverse AND invert**: `1 - actions[T-1-t][6]` |

**Key insight**: Since actions are velocity commands, time reversal requires **negating** the velocity (moving backwards along the same trajectory). For gripper: if the original closes the gripper, the reversed version opens it.

### 7.4 Verification

After reversal, verify:
1. **Position reconstruction**: Integrating reversed velocities from the reversed starting pose should trace the original trajectory backwards
2. **Boundary match**: reversed[0].tcp_pos == original[-1].tcp_pos
3. **Gripper logic**: If original picks up (closes gripper), reversed places down (opens gripper)

### 7.5 Usage

```bash
python scripts/scripts_task_inovo/2_time_reverse.py \
    --input data/inovo_data/0209_tower_boby \
    --output data/inovo_data/tower_boby_A \
    --verify \
    --new_language "Pick up the purple disc from the sponge and place it on the pole."
```

**Output**: Same directory structure as input, with reversed episodes.

## 8. Script 3: Convert to LeRobot Format

### 8.1 Purpose

Convert the RoboKit per-frame NPZ format into [LeRobot v3.0 dataset format](https://huggingface.co/docs/lerobot) for policy training.

### 8.2 Planned Script: `3_convert_to_lerobot.py`

### 8.3 Format Mapping

| Source (RoboKit NPZ) | Target (LeRobot) | Notes |
|----------------------|-------------------|-------|
| `primary_rgb` (480×848) | `observation.image` (H×W) | Resize to training resolution (e.g., 240×320) |
| `gripper_rgb` (480×848) | `observation.wrist_image` (H×W) | Resize to same resolution |
| `robot_obs[0:6]` (tcp pose) | `observation.state` (6D or 7D) | Optionally include gripper_width |
| `robot_obs[6]` (gripper width) | `observation.state[6]` | If `--include_gripper` |
| `actions[0:6]` (velocity) | `action[0:6]` | **Need to decide: keep as velocity or convert to delta pose** |
| `actions[6]` (gripper) | `action[6]` | Binary gripper command |

### 8.4 Action Format Decision

**Option A: Keep as velocity commands** (native format)
- Pro: No conversion errors, faithful to original data
- Con: Different from Piper pipeline which uses delta pose; may need different action post-processing at inference

**Option B: Convert to delta pose** (`delta_pose[t] = obs[t+1].tcp_pose - obs[t].tcp_pose`)
- Pro: Consistent with existing Piper training scripts; reuse more code
- Con: Introduces conversion artifacts

**Recommendation**: Start with **Option B** (delta pose) for maximum code reuse with existing training infrastructure. If results are poor, fall back to Option A.

### 8.5 Usage

```bash
# Convert Task B (original)
python scripts/scripts_task_inovo/3_convert_to_lerobot.py \
    --input data/inovo_data/0209_tower_boby \
    --output runs/inovo_B/lerobot_dataset \
    --image_size 240 320 \
    --include_gripper \
    --action_format delta_pose

# Convert Task A (reversed)
python scripts/scripts_task_inovo/3_convert_to_lerobot.py \
    --input data/inovo_data/tower_boby_A \
    --output runs/inovo_A/lerobot_dataset \
    --image_size 240 320 \
    --include_gripper \
    --action_format delta_pose
```

## 11. Script 6: Iterative DAgger Pipeline

### 11.1 Purpose

Same as `scripts/scripts_pick_place/run_ablation.sh` — iterative evaluate → aggregate → train loop with parallel fair testing.

### 11.2 Prerequisites

This step requires a working evaluation environment (sim or real). The DAgger pipeline:

1. **Evaluate** current policies: Run 50 A-B cycles collecting successful rollout data
2. **Fair Test** (parallel): Run 50 independent episodes per task with full reset
3. **Aggregate**: Merge successful rollouts into training dataset
4. **Train**: Finetune policies for 5000 additional steps
5. **Repeat**: 10 iterations → 11 fair-test data points

## 12. Key Differences from Piper

| Aspect | Piper | Inovo |
|--------|-------|-------|
| **Data origin** | Collected ourselves via PS5 teleoperation | Pre-existing TCL team data |
| **Data format** | Our tar.gz episode archives (NPZ + PNG dirs) | RoboKit per-frame NPZ (pickled + JPEG-encoded) |
| **Action type** | Delta pose (position change per step) | Velocity command (m/s, rad/s at ~30 Hz) |
| **Orientation** | Euler angles (radians), directly in action | Euler angles (radians), in `robot_obs[3:6]` |
| **Image resolution** | 640×480 (fixed + wrist) | 848×480 (primary + gripper) |
| **Robot DoF** | 6-DoF + parallel gripper | 6-DoF + parallel gripper |
| **Evaluation** | Isaac Lab (Franka sim) | TBD (Inovo sim or real) |
| **Gripper state** | Continuous [0, 1] | Continuous width [0, 1] + binary action {0, 1} |
| **Policy architecture** | Diffusion (U-Net) or DiT Flow | DiT Flow (primary choice) |
| **Coordinate frame** | Robot base frame | Robot base frame |
| **Data collection control** | PS5 controller or FSM scripted | Unknown (TCL team collected) |

### 12.1 Action Conversion Note

The Inovo data stores **velocity commands**: `actions = [vx, vy, vz, v_roll, v_pitch, v_yaw, gripper]`

To convert to **delta pose** (compatible with existing training code):
```python
dt = 1.0 / 30.0  # ~30 Hz data collection rate
delta_pose = actions[0:6] * dt  # velocity × dt = position change
gripper = actions[6]            # keep as-is
```

Or equivalently, compute delta pose directly from observations:
```python
delta_pose[t] = robot_obs[t+1][0:6] - robot_obs[t][0:6]
```

The second approach is more robust as it reflects the actual motion rather than the commanded velocity.

## 14. Open Questions & Risks

### 14.1 Critical Questions

1. **Action format**: Should we train on velocity commands (native) or convert to delta pose (Piper-compatible)? Delta pose allows maximum code reuse but introduces conversion artifacts.

2. **Evaluation environment**: Do we have an Inovo arm simulation (Isaac Lab USD asset)? If not, real hardware is needed for evaluation, which complicates the DAgger loop.

3. **Task reversibility**: Is the tower body task cleanly reversible? Picking from a pole (requires precise alignment) may be harder than placing on a sponge (more tolerance). The reversed Task A might be inherently more difficult.

4. **Gripper behavior during reversal**: The original data uses gripper closing to grasp from the pole. When reversed, the gripper opening should release onto the pole. Need to verify the gripper timing makes physical sense after reversal.

5. **Image aspect ratio**: Original images are 848×480 (≈16:9). Resizing to 240×320 (4:3) changes the aspect ratio. Should we center-crop or letterbox instead?

### 14.2 Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Velocity → delta pose conversion loses information | Training quality drop | Fall back to native velocity training |
| No simulation environment for Inovo | Cannot run DAgger loop autonomously | Evaluate on real hardware manually |
| Tower body task is not reversible | Rev2Fwd method doesn't apply | Try simpler pick-place datasets instead |
| Image quality issues (JPEG compression in NPZ) | Visual artifacts in training | Check image quality; re-encode if needed |
| Data collected by others may have inconsistent quality | Some episodes unusable | Filter bad episodes in Step 0 |
| 848×480 resolution is wider than Piper's 640×480 | Model may not handle well | Center-crop to 640×480 before resize |

## 15. Bug Fix Log

Scripts were audited against this README on 2026-03-05. The following bugs were found and fixed:

### 15.1 \[CRITICAL\] Script 2: 文件名映射导致时间反转完全失效

**文件**: `2_time_reverse.py` 保存逻辑

**问题**: `reversed_frames[i]` 被保存到文件名 `npz_files[T-1-i].name`（即倒数第 `i` 个原始文件名）。但下游所有脚本（Script 1/3/5）都用 `sorted(glob("*.npz"))` 按文件名排序加载帧，这意味着保存后再加载时帧顺序**恢复成了原始顺序**，时间反转完全无效。

**修复**: 改为 `npz_files[i].name`——让第 `i` 个反转帧使用第 `i` 个（时间最早的）文件名，这样 `sorted()` 加载后的顺序就是正确的反转帧顺序。

### 15.2 \[CRITICAL\] Script 2: 不存在的 import 导致脚本崩溃

**文件**: `2_time_reverse.py` dt 估计逻辑

**问题**: 在不指定 `--dt` 参数时，代码执行到：
```python
from scripts.scripts_task_inovo import _estimate_dt  # 此模块/函数不存在
```
由于这行 import 不在 `try/except` 内，会直接抛出 `ImportError`，脚本无法运行。

**修复**: 删除了这行死代码，让下方已有的内联时间戳估计逻辑直接执行。

### 15.3 \[MODERATE\] Script 2: 速度动作索引偏移一位

**文件**: `2_time_reverse.py` → `reverse_episode()` 函数

**问题**: 原代码：
```python
actions_new[:6] = -src["actions"][:6]  # src = frames[T-1-t]
```
即 `rev_actions[t] = -orig_actions[T-1-t]`。但 `orig_actions[T-1-t]` 描述的转移是 `obs[T-1-t] → obs[T-t]`，而反转后第 `t` 帧需要的转移是 `obs[T-1-t] → obs[T-2-t]`（即 `rev_obs[t] → rev_obs[t+1]`），正确应取 `-orig_actions[T-2-t]`。

Piper 的参考实现（`3_make_forward_data.py`）通过从观测差值重算动作 `ee_pose_rev[t+1] - ee_pose_rev[t]` 来避免此问题。

**影响**: 使用 `--action_format velocity` 训练时速度标签全部偏移一帧。默认的 `delta_pose` 模式不受影响（它从观测差值重算）。

**修复**: 改为使用 `frames[T-2-t]["actions"]` 作为动作源，最后一帧使用零动作兜底。

### 15.4 \[MODERATE\] Script 5: 在线评估接口语义错误 + gripper 逻辑丢失

**文件**: `5_eval_policy.py` → `InovoRobotInterface` 类 + `run_online_episode()`

**问题**: 在线评估采用积分控制（将预测的 delta 累加到首帧位姿上），实际发送给机器人的是**绝对目标位姿**，但接口方法叫 `send_action()` 且文档声称接收 delta 动作，会误导实现者。此外，编辑过程中 gripper forcing 逻辑（夹紧强化 + 末尾强制张开）意外丢失，导致 `gripper_val` 变量未定义。

**修复**:
- 方法重命名为 `send_pose_target(target_pose, gripper)`，更新文档说明参数是绝对位姿
- 恢复 gripper forcing 逻辑（预测值 < 0.9 时强制为 0.3，最后 50 步强制张开）
